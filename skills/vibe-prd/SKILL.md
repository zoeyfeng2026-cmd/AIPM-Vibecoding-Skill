# Vibe Coding PRD — 写给 AI 看的产品需求文档

## 这是什么

Vibe Coding PRD 是一套面向 AI 执行的产品需求文档方法论。传统 PRD 写给人看（可以模糊、口头补充上下文），Vibe Coding PRD 写给 AI 看——必须精确、自包含、可执行。

使用者是非技术背景的产品经理，通过自然语言指挥 AI 工具（Claude Code、Cursor 等）写代码来构建产品。

**核心原则：描述怎么做 + 不做什么 + 怎么验证做对了。**

跨平台通用：Web App、iOS、小程序、Android、桌面端。

## 触发条件

本 Skill 仅在以下场景激活：

- 用户输入 `/vibe-prd`
- 用户明确提到"写 vibe coding 的 PRD"或"给 AI 写的 PRD"
- 用户说"用 vibe coding 做一个 XX 产品"并需要先写需求文档

**不激活的场景**：用户只是提到"PRD"但不是给 AI 执行用的、用户在做传统开发流程、用户在讨论产品策略而非要落地执行。

## 撰写流程（4 个阶段）

PRD 撰写不是直接填模板。必须按以下 4 个阶段顺序执行，每个阶段有明确的确认环节。

### 阶段 1：需求理解与确认

**目标**：确保 AI 完全理解用户想做什么、为什么做、边界在哪。

**执行方式**：AI 通过提问收集信息，然后输出一份「需求摘要」让用户确认。

**AI 需要问清楚的问题**（至少覆盖以下维度）：

```
1. 产品是什么？（一句话描述）
2. 为谁做的？（目标用户是谁，他们的核心痛点）
3. 核心功能有哪些？（列 3-5 个关键功能）
4. 不做什么？（明确排除的范围）
5. 目标平台？（Web / iOS / 小程序 / Android / 桌面端）
6. 有没有参考产品或设计偏好？
```

**产出**：一份简短的需求摘要（不超过半页），格式如下：

```
我理解你的需求是：
- 产品：[一句话]
- 用户：[谁]
- 核心功能：[列表]
- 不做什么：[列表]
- 平台：[目标平台]

对吗？有遗漏或理解偏差吗？
```

**确认规则**：用户确认后才进入阶段 2。用户说"不对"或"漏了 XX"，AI 修改摘要后再次确认。

---

### 阶段 2：产品架构设计与确认

**目标**：在写详细 PRD 之前，先和用户对齐产品的宏观结构。

**执行方式**：AI 基于阶段 1 的确认结果，设计产品架构方案，让用户确认。

**AI 输出的内容**：

```
产品架构方案：

1. 页面结构
   - 有哪些页面？每个页面做什么？
   - 页面之间的跳转关系

2. 数据核心
   - 有哪些核心实体？（不需要详细字段，只需要概念层）
   - 实体之间的关系（一句话描述）

3. 技术方案
   - 推荐的技术栈（一句话理由）
   - 部署方式

4. 外部依赖
   - 需要调用哪些外部服务？
   - 有没有可选的替代方案？

你确认这个方向吗？有需要调整的地方吗？
```

**确认规则**：用户确认后才进入阶段 3。用户可以要求调整任何部分（换技术栈、加减页面、调整数据结构等）。

---

### 阶段 3：PRD 撰写（5 个模块）

**目标**：基于确认后的需求和架构，撰写完整的 5 模块 PRD。

**执行方式**：逐模块撰写，每个模块写完后让用户确认，再进入下一个。

```
模块 1（产品意图）→ 用户确认 →
模块 2（技术约束）→ 用户确认 →
模块 3（数据与接口）→ 用户确认 →
模块 4（交互与设计）→ 用户确认 →
模块 5（验证与边界）→ 用户确认
```

**每个模块的确认格式**：

```
模块 X 已完成，核心内容：
- [关键决策 1]
- [关键决策 2]
- [关键决策 3]

需要修改吗？没问题的话进入下一个模块。
```

**用户可以随时说**：
- "这个模块跳过" → AI 使用阶段 2 的信息自动填充，继续下一个
- "这里要改" → AI 修改后再次确认
- "所有模块一起写吧" → AI 一次性输出 5 个模块，最后统一确认

---

### 阶段 4：二次确认与交付

**目标**：最终检查一致性，生成所有文件。

**执行方式**：

1. AI 对 5 个模块做一致性检查：
   - 模块 1 的"不做什么"和模块 4 的页面、模块 5 的"不做什么"是否矛盾
   - 模块 2 的技术栈和模块 3 的数据实现方式是否匹配
   - 模块 4 的所有页面在模块 3 中是否有对应的 API

2. AI 输出一致性检查结果：
   ```
   一致性检查通过的项：✓
   需要你确认的矛盾点：
   - [如果有矛盾，列出具体问题]
   ```

3. 用户最终确认后，AI 生成文件：
   ```
   项目根目录/
   ├── PRD/
   │   ├── 01-product-intent.md
   │   ├── 02-tech-constraints.md
   │   ├── 03-data-interfaces.md
   │   ├── 04-interaction-design.md
   │   └── 05-verification.md
   └── CLAUDE.md    ← 从 PRD 浓缩的 AI 执行指南
   ```

## 两种使用方式

### 方式 A：斜杠命令（交互式引导，推荐）

用户输入 `/vibe-prd`，AI 按上述 4 个阶段执行。

### 方式 B：文档模板（自主填写）

提供 5 个模板文件（见 `templates/` 目录），PM 按照模板中的规则自行填写，填完后让 AI 读取执行。AI 读完后仍需走阶段 4 的一致性检查。

模板中每个字段都有：
- **填写规则**：怎么写才合格
- **好的写法**：格式示范（展示写法标准，不暗示具体产品）
- **差的写法**：常见的模糊写法（要避免）

## 5 个必写模块

### 模块 1: 产品意图（Product Intent）

**文件**：`PRD/01-product-intent.md`

**定义**：产品是什么、为谁做、核心价值、品牌调性、什么不该做。

**为什么必须有**：AI 没有项目背景。当实现中遇到歧义（该不该加某个功能、优先级怎么排、风格怎么选），AI 会回到这里找决策锚点。没有这一章，AI 会在 50 个选择面前随机选。

**填写规则：**

| 字段 | 怎么写 | 好的写法 | 差的写法 |
|------|--------|---------|---------|
| 产品定位 | 一句话说清"这是什么产品"，含形态+功能+价值 | "个人使用的 Web 应用，[核心功能]，让用户获得 [具体结果]" | "一个 AI 聊天工具" |
| 目标用户 | 具体到一类人 + 核心诉求 | "[身份]，希望 [具体目标]" | "所有想用 AI 的人" |
| 核心价值 | 用户获得的具体结果，2-3 条 | "功能 A → 用户得到 B" | "很好用" |
| 品牌调性 | 2-3 个关键词 + 风格参考 | "关键词 A、关键词 B。参考 [具体产品或感觉]" | "好看一点" |
| 不做什么 | 明确排除范围 | "不做 X、不做 Y、不做 Z" | 不写 |

**必须包含的章节：**
1. 产品概述（定位 + 用户 + 价值）
2. 品牌调性（关键词 + 风格参考）
3. 不做什么（排除清单）

### 模块 2: 技术约束（Technical Constraints）

**文件**：`PRD/02-tech-constraints.md`

**定义**：锁定技术栈、部署方式、环境变量、第三方服务。

**为什么必须有**：AI 有"自作主张"倾向——你不指定框架它可能用 Vue 而不是 React，不指定数据库它可能上 PostgreSQL。技术约束限定选择空间，减少返工。

**填写规则：**

| 字段 | 怎么写 | 好的写法 | 差的写法 |
|------|--------|---------|---------|
| 技术栈 | 指定每个层级的技术 + 版本 | "框架名 版本号 (特性) + 语言 + 样式方案" | "用 React" |
| 选型理由 | 每个选择一句话说为什么 | "选择 X：一句话理由" | 不写 |
| 部署方式 | 明确本地/云/容器 | "本地 [命令]，生产 [平台]" | "部署到服务器" |
| 环境变量 | 列出所有必须的 key | "KEY_NAME：用途说明" | "配一下 key" |
| 第三方服务 | 列出所有外部依赖 + 用途 + 接入方式 | "服务名（用途，接入方式）" | "用 AI 服务" |
| 禁止项 | 明确不能用的技术 + 理由 | "不引入 X（理由）" | 不写 |

**必须包含的章节：**
1. 技术栈表格（层级 → 技术 → 版本 → 理由）
2. 部署方案（本地 + 生产环境）
3. 环境变量清单
4. 禁止项

### 模块 3: 数据与接口（Data & Interfaces）

**文件**：`PRD/03-data-interfaces.md`

**定义**：数据模型（表/实体/字段）、实体间关系、API 接口定义。

**为什么必须有**：数据模型是产品骨架，接口是模块间契约。定义清楚了，UI 和逻辑都是"围绕数据的操作"。定义不清楚，AI 会在不同文件里重复定义类型、字段名不一致、漏掉关联。

**填写规则：**

| 字段 | 怎么写 | 好的写法 | 差的写法 |
|------|--------|---------|---------|
| 实体定义 | 每个字段：名称、类型、说明 | "id: TEXT PK, name: TEXT 用途说明, ..." | "有 id 和 name" |
| 关系定义 | 用文字或 ER 图描述 | "A ↔ B 通过关联表 C 多对多关联" | "有关联" |
| ID 命名规则 | 统一前缀 | "前缀-${uuid}" | 不写 |
| API 定义 | 方法 + 路径 + 请求体 + 响应体 | "POST /api/xxx, Body: {字段}, Response: {结构}" | "有个接口" |
| 流式协议 | 定义事件类型和顺序 | "SSE: event1 → event2* → done" | "流式返回" |

**必须包含的章节：**
1. ER 图或关系描述
2. 每个实体的完整字段表
3. API 接口列表（方法、路径、入参、出参）
4. 流式接口的事件定义（如有）

### 模块 4: 交互与设计（Interaction & Design）

**文件**：`PRD/04-interaction-design.md`

**定义**：页面结构、用户操作流程、视觉规范。

**为什么必须有**：AI 需要知道"点按钮发生什么"、"列表为空显示什么"、"加载中什么状态"。没有交互流程，AI 自编一套前后不一致。没有设计规范，每个页面长得不一样。

**填写规则：**

| 字段 | 怎么写 | 好的写法 | 差的写法 |
|------|--------|---------|---------|
| 页面清单 | 列出所有页面 + 路径 + 一句话说明 | "/path 页面名：一句话说明" | "几个页面" |
| 页面布局 | ASCII 线框图或结构描述 | "上 [顶栏]，左 [侧边栏]，右 [内容区]" | "有侧边栏" |
| 核心流程 | 分步骤写每个关键操作 | "1. 触发条件 → 结果 2. ..." | "用户可以操作" |
| 设计系统 | 颜色/字体/间距/圆角，精确到值 | "Primary: #HEX, 字体: 名称, 圆角: Npx" | "用暖色调" |
| 状态定义 | 空状态、加载状态、错误状态 | "组件名：空状态表现 / 加载表现 / 错误表现" | 不写 |
| 响应式 | 桌面/移动端差异 | "桌面端 [布局]，移动端 [布局]" | "适配手机" |

**必须包含的章节：**
1. 页面清单 + 路由
2. 页面布局（至少描述结构）
3. 核心操作流程（分步骤）
4. 设计规范（颜色/字体/间距精确值）
5. 状态定义（空/加载/错误）

### 模块 5: 验证与边界（Verification & Boundaries）

**文件**：`PRD/05-verification.md`

**定义**：成功标准、测试流程、兜底策略、性能限制、不做什么。

**为什么必须有**：传统 PRD 只说"做什么"，不说"怎么验证做对了"和"失败了怎么办"。AI 会省略错误处理、跳过边界条件。这一章强制 AI 在写代码时同时写兜底逻辑和验证手段。

**填写规则：**

| 字段 | 怎么写 | 好的写法 | 差的写法 |
|------|--------|---------|---------|
| 成功标准 | 可执行的端到端流程 | "1. 启动 → 看到 X 2. 操作 → 看到 Y 3. ..." | "能用就行" |
| 兜底策略 | 每个外部依赖失败的处理 | "服务名 超时 → 处理方式，不影响其他功能" | "出错就报错" |
| 性能限制 | 具体数字 | "单条最大 N 字符，列表最多 N 条，文件最大 N MB" | "不要太慢" |
| 不做什么 | 排除范围（与模块 1 互补） | "V1 不做 X、不做 Y" | 不写 |
| 降级方案 | 核心依赖不可用时的替代 | "功能 X 不可用 → 降级表现，其他功能正常" | 不写 |

**必须包含的章节：**
1. 成功标准（端到端验证流程）
2. 兜底策略（每个外部依赖的失败处理）
3. 性能限制（具体数字）
4. V1 不做什么
5. 降级方案
